# Milestone v1.2 Roadmap: LSP Optimization

**Created:** 2026-04-29  
**Status:** Locked roadmap - ready for Phase 10 discuss  
**Codex thread:** `019dd836-27d3-7c51-ad48-d06a7851133b`

## Scope

v1.2 optimizes the built-in dbee LSP module without changing its core architecture.

Keep:
- In-process LSP server via `vim.lsp.start({ cmd = server.create(cache) })`.
- Singleton-per-current-connection model.
- Stale-while-revalidate disk cache pattern.
- Handler-owned lifecycle, root epoch, and drawer/LSP single-flight contracts from v1.1.

Do not:
- Pivot to standalone `sqls`.
- Move the LSP out-of-process.
- Reopen v1.0/v1.1 locked decisions D-01..D-118.
- Fork Phase 9 perf infrastructure.

## Research Inputs

Two independent research outputs converged:
- `.planning/phases/10-lsp-optimization/10-RESEARCH.md`
- `.planning/research/v12-lsp-opus-research.md`

Shared findings:
- The architecture is sound and should be preserved.
- The P0 latency hazard is synchronous `handler.connection_get_columns()` inside the completion request path.
- Lower-tier hazards include full-buffer diagnostics on every `didChange`, repeated schema/table scans, unbounded column-cache growth, synchronous startup disk reads, and silent non-atomic cache writes.
- Real correctness bugs exist in table-reference diagnostics: multi-line references, JOIN position calculation, and schema-qualified validation.
- LSP lacks a deterministic headless perf harness comparable to Phase 9.
- Neovim 0.12 alignment is mostly hygiene: `vim.loop` to `vim.uv`, explicit floor/docs, optional future LSP config migration.

## Roadmap Shape

v1.2 is two mandatory phases plus one conditional phase.

| Phase | Name | Status | Rationale |
| --- | --- | --- | --- |
| 10 | LSP Perf Harness | Mandatory | Land measurement first so optimization has baseline evidence and regression gates. |
| 11 | LSP Perf Optimization And Correctness | Mandatory | Remove the P0 completion blocker, reduce typing-path work, and fix diagnostics bugs. |
| 12 | LSP Feature Gap Closure | Conditional | Add hover/resolve/code-action/symbol features only if Phase 10+11 finish with budget headroom. |

Ordering is locked: **harness -> optimize -> conditional features**.

## Fork Locks

These fork decisions are locked for v1.2 and should be carried into Phase 10/11 context. Phase decision numbering still starts at **D-119** during `$gsd-discuss-phase 10`.

| Fork | Locked decision | Rationale |
| --- | --- | --- |
| Async runtime | Adopt `nvim-nio` as a peer dependency; use it to orchestrate futures over the existing `connection_get_columns_async` / `structure_children_loaded` surface, not by wrapping synchronous `connection_get_columns()`. | Cleaner future/coroutine orchestration than hand-rolled Lua, while preserving the existing async endpoint that actually moves column fetching off the completion path. |
| Cache-miss policy | Return cached completion results immediately, warm missing columns asynchronously, and set `isIncomplete = true` only when a cache miss is actively in flight. | Matches the converged research and avoids blocking typing. |
| Diagnostics policy | Default to 250ms debounced diagnostics on `didChange`; add plugin config for save-only or off. | Preserves current behavior intent while bounding typing-path work. |
| Multi-connection LSP | Stay singleton-per-current-connection for v1.2. | Multi-client/buffer-pinned LSP is a non-trivial model change and not needed for the perf milestone. |
| Neovim floor | Bump the supported floor to Neovim `0.12.x` for v1.2. | Phase 9 already established 0.12.x for perf evidence; v1.2 LSP gates should align with that floor. |
| LRU bound | Cap in-memory `SchemaCache.columns` at 500 tables; disk cache remains unbounded except startup pruning of files older than 30 days. | Conservative memory bound while preserving stale-while-revalidate behavior. |
| Feature priority | Defer LSP feature additions to conditional Phase 12. | Perf and correctness are the core milestone value. |
| Primary completion surface | Treat dbee built-in LSP as canonical; keep external `cmp-dbee` working and documented as an alternate/custom-stack path. | Built-in LSP is the maintained in-repo path, but existing completion-stack users should not be broken. |

## Phase 10: LSP Perf Harness

**Goal:** Stand up a deterministic headless LSP perf harness before changing behavior.

**Depends on:** Phase 9 infrastructure and v1.1 lifecycle locks.

**Risk:** Low. Measurement-only.

**Success criteria:**
1. `make perf-lsp` runs locally and uses the same bootstrap pattern as `make perf`.
2. LSP perf bootstrap reuses `ci/headless/perf_bootstrap.mk`, pinned `benchmark.nvim`, pinned `profile.nvim`, and the threshold source-of-truth pattern in `ci/headless/perf_thresholds.lua`.
3. CI has macOS and Linux advisory LSP perf lanes with grep-friendly markers, median/p95 reporting, and artifact upload.
4. Scenarios include cold LSP start, table completion with 100/1000/10000 tables, cold `column_of_table` completion, diagnostics over 100/1000/10000-line buffers, and alias parsing scaling.
5. `bench.lua` remains an interactive probe, but relevant steps are available as deterministic headless benchmark scenarios.

**Research bullets:**
- Reuse Phase 9's advisory-to-blocking policy: four weeks at >=95% pass rate before promotion.
- Keep threshold tables per platform; Linux and macOS are co-equal advisory lanes.
- Avoid live database dependencies. Synthetic caches and fake handler surfaces are the blocking corpus.
- Record Neovim version, platform, sample count, median, and p95 in the summary artifact.

## Phase 11: LSP Perf Optimization And Correctness

**Goal:** Remove the P0 and lower-tier perf hazards while fixing diagnostics correctness bugs surfaced by research.

**Depends on:** Phase 10.

**Risk:** Medium-high. Touches completion, diagnostics, and cache behavior.

**Success criteria:**
1. Cold-cache alias/table dot completion never blocks on synchronous `connection_get_columns()`.
2. Column cache misses dedupe in flight per `(conn_id, schema, table, materialization, root_epoch)` and complete through the async column event path.
3. `SchemaCache` precomputes sorted completion arrays and case-folded lookup indexes, invalidated only on cache mutation.
4. `SchemaCache.columns` has a 500-table in-memory LRU and startup disk-cache pruning for files older than 30 days.
5. `didChange` diagnostics are debounced by default at 250ms, with save-only/off config options.
6. Diagnostics validate multi-line `FROM`/`JOIN`, compute JOIN columns correctly for multiple joins on a line, and use schema-aware validation for `schema.table`.
7. Disk cache writes use temp-file-plus-rename and log read/write failures at appropriate debug/warn levels.
8. `bench.lua` uses `vim.uv` instead of `vim.loop`.
9. Phase 10 perf scenarios show no regression outside explicitly accepted tradeoffs.

**Research bullets:**
- Preserve Phase 7 handler-owned root epoch and single-flight semantics.
- Use existing `Handler:connection_get_columns_async()` and `structure_children_loaded` rather than inventing a second async metadata transport.
- Keep disk cache stale-while-revalidate; do not turn cache misses into hard errors.
- Make diagnostics behavior configurable without changing adapter execution diagnostics from Phase 5.

## Phase 12: LSP Feature Gap Closure

**Goal:** Expose high-value LSP features that can be supported truthfully from schema/cache state.

**Depends on:** Phase 11.

**Status:** Conditional. Decide at Phase 11 ship. If Phase 10+11 land with budget headroom and no major regressions, execute Phase 12 in v1.2; otherwise defer to v1.3.

**Risk:** Low-medium. Additive features, no architecture pivot.

**Candidate scope:**
- Completion item `data` plus `completionItem/resolve` for lazy details.
- Hover for table/column metadata.
- Code actions for refresh/reload on stale or unknown-schema diagnostics.
- Document/workspace symbols for cached schema objects.

**Deferred explicitly:**
- Semantic tokens; needs treesitter or a proper parser.
- Inlay hints; UX uncertain.
- `vim.lsp.config()` migration; mechanical and low value for this milestone.
- Multi-client/per-buffer connection model.
