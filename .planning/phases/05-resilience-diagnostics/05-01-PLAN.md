<listener_invariants>
## Listener Ordering & Invariants

- The reconnect manager listener does not mutate diagnostics state. Its scope is limited to prompt ownership, replay ownership, and retry-registry lifecycle.
- Diagnostic clearing on retry happens when the new call enters its normal execution lifecycle, not inside reconnect callbacks.
- Multiple `call_state_changed` listeners run within the same scheduled dispatch tick. Every reconnect listener branch must be O(1) per event and must not block on UI or network work outside the single `vim.ui.select` prompt handoff.
- No branch of `handle_call_state_changed()` may walk the full retry registry. Lookups are O(1), and any multi-entry cleanup or connection-identity rewrite must use `call_ids_by_conn[conn_id]` so cost is scoped to one connection.
- Connection-rewrite listeners are keyed. Duplicate `register_connection_rewritten_listener(subscriber_key, ...)` attempts must fail loudly instead of silently double-subscribing.
- Prompt confirm handlers must re-read the live connection key (`effective_conn_id`), `episodes[effective_conn_id]`, and `calls_by_id[id]` at invocation time. They must not capture an episode table or retry target when the prompt opens because superseding disconnects, episode resets, or connection-identity rewrites may replace and forget them before the user answers.
</listener_invariants>

         local kind = data.call.error_kind

           end

           return

           return
         end

         if (state == "executing_failed" or state == "retrieving_failed" or state == "archive_failed")

           return

           return

           retired_by_retry[data.call.id] = nil
           return

         episodes[effective_conn_id] = episode

             return
           end

           if not call_id then
             utils.log("warn", ('No target call for reconnect on "%s"'):format(conn_name))
             return

           if not latest_meta then

             conn_name = new_conn_name,
