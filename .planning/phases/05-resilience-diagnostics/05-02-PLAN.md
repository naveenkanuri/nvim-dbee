Recommended new diagnostics module contract:
```lua
local diagnostics = require("dbee.ui.editor.diagnostics")

---@class dbee_diag_ctx
---@field resolved_query string?
---@field start_line integer
---@field start_col integer

---@class dbee_diag_result
---@field line integer
---@field col integer
---@field message string
---@field severity integer

---@class dbee_diag_parser_opts
---@field sql boolean
---@field aliases string[]?

---@param adapter string
---@param parser fun(err_msg: string, ctx: dbee_diag_ctx): dbee_diag_result?|nil
---@param opts dbee_diag_parser_opts
function diagnostics.register_parser(adapter, parser, opts) end

---@param adapter string
---@return boolean
function diagnostics.is_sql_adapter(adapter) end

---@param adapter string
---@param err_msg string
---@param ctx dbee_diag_ctx
---@return dbee_diag_result|nil
function diagnostics.build_diagnostic(adapter, err_msg, ctx) end
```

The registration API itself must declare SQL-ness and aliases so there is no separately-maintained `sql_adapters` set, and it must build the alias reverse map eagerly:
```lua
diagnostics.register_parser("postgres", postgres_parser, { sql = true, aliases = { "pg", "postgresql" } })
diagnostics.register_parser("mysql", mysql_parser, { sql = true })
diagnostics.register_parser("sqlite", nil, { sql = true, aliases = { "sqlite3" } }) -- fallback only in Phase 5
diagnostics.register_parser("sqlserver", sqlserver_parser, { sql = true, aliases = { "mssql" } })
diagnostics.register_parser("oracle", oracle_parser, { sql = true })
diagnostics.register_parser("clickhouse", nil, { sql = true })
```

Connection-scoped namespace helpers in `EditorUI`:
```lua
---@field private diag_ns_by_conn table<connection_id, integer>
---@field private diag_bufs_by_conn table<connection_id, table<integer, true>>
---@field private note_count_by_conn table<connection_id, integer>

---@param conn_id connection_id
---@return integer
function EditorUI:get_diag_namespace(conn_id) end

---@param conn_id connection_id
---@param bufnr integer?
function EditorUI:prune_diag_namespace(conn_id, bufnr) end
```

`prune_diag_namespace(conn_id, bufnr)` semantics:
- when `bufnr` is provided, reset only that tracked buffer and update `diag_bufs_by_conn[conn_id]`
- when `bufnr` is omitted, iterate only `diag_bufs_by_conn[conn_id]` and reset all tracked diagnostic buffers for that connection
- no `bufnr = 0` sentinel behavior is allowed in Phase 5 because it only targets the current buffer

Reconnect-local cleanup contract from `05-01`:
```lua
local reconnect = require("dbee.reconnect")
reconnect.register_connection_rewritten_listener("editor-diagnostics", function(old_conn_id, new_conn_id) end)
```

Listener semantics:
- the payload scope is exactly `(old_conn_id, new_conn_id)`
- diagnostics cleanup may touch only those two connection IDs plus buffers already tracked under `diag_bufs_by_conn`
- duplicate subscriber keys must raise a clear error instead of silently double-subscribing
- `current_connection_changed` is intentionally not part of the pruning contract because `CONN-01` may emit it internally while preserving user context per D-07
