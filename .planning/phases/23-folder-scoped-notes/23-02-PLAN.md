# 23-02 - Migration And Folder Namespace Lifecycle

<objective>
Migrate legacy global notes into folder namespaces once, prevent `global` recreation, and make folder create/delete/rename/move operations maintain note namespace semantics.
</objective>

<threat_model>
- Risk: concurrent nvim instances both migrate and delete/copy the same files. Mitigation: atomic lock directory plus post-lock sentinel recheck.
- Risk: folder deletion permanently deletes notes before folder removal fails. Mitigation: stage namespace directory and restore on handler failure.
- Risk: unsafe folder IDs from manual sidecar edits escape notes root. Mitigation: validate folder IDs before namespace filesystem operations.
- Risk: migration partial failure deletes legacy global. Mitigation: clone first, backup/rename second, sentinel last.
</threat_model>

<must_haves>
- Migration is first-launch post-Phase-23 and idempotent.
- Migration clones, not moves, each legacy global note into every existing folder namespace.
- Original `notes/global/` path is removed after successful clone/backup.
- Backup remains available until manual cleanup.
- Folder created after sentinel gets an empty namespace only.
- Folder rename never moves a namespace directory.
- Folder delete confirmation mentions note deletion.
- Editor startup never calls `namespace_create_note("global", "welcome")` after Phase 23 migration.
</must_haves>

<action>
Add EditorUI folder namespace helpers and Phase 23 migration.

Required helpers:

1. `folder_namespace_id(folder_id)` returns exactly `"folder:" .. folder_id`.
2. Validate folder IDs before filesystem use. Accept ONLY when `folder_id:match("^folder_[A-Za-z0-9]+$")` is non-nil. The regex IS the gate. NO additional substring blacklist. Reject all other inputs including: empty string, nil, `folder` (no underscore-anything), `Folder_abc` (case-sensitive), `folder_a/b`, `folder_a%00b` (NUL byte), `folder_a.b`, `..`, `folder_-x`.
3. `folder_namespace_dir(folder_id)` returns `self.directory .. "/" .. folder_namespace_id(folder_id)`.
4. `ensure_folder_namespace(folder_id)` creates the directory with `vim.fn.mkdir(dir, "p")`.
5. `list_existing_folder_namespaces()` enumerates folder IDs from every folder-capable source via Handler/Source APIs.

Required migration:

1. Run from a post-handler-bootstrap hook in `dbee.setup()` AFTER all `FileSource:_ensure_folders_loaded()` calls complete and ALL folder-capable sources are in `loaded_ok` state. Migration MUST NOT run during `EditorUI:new()`.
2. Use sentinel path `self.directory .. "/.notes-migration-v1"`.
3. Use lock directory path `self.directory .. "/.phase23-migration.lock"` and acquire it FIRST with `vim.loop.fs_mkdir(path, 448)`, handling EEXIST as "another instance holds the lock". Stale lock age threshold is exactly 5 minutes.
4. After acquiring lock, re-check sentinel. If `notes/.phase23-recovery-needed` exists from a previous sentinel-write failure and folder clones are present, write the missing sentinel atomically and remove the recovery marker.
5. AFTER lock acquisition AND AFTER sentinel re-check, enumerate folders via `list_existing_folder_namespaces()`. Folders created by other instances AFTER this enumeration are intentionally not seeded.
5a. Before any notes data filesystem mutation after lock/sentinel handling:
   - validate that every folder-capable source has `_folders_load_state == "loaded_ok"`.
   - if ANY source has `load_failed` OR returns no folders despite the source claiming folder-capability, ABORT migration.
   - do NOT write sentinel.
   - do NOT delete `notes/global/`.
   - do NOT create backup.
   - emit `vim.notify('dbee: migration aborted — folder source load failed; fix sidecar before retrying', vim.log.levels.WARN)`.
5b. Duplicate folder ID guard:
   - enumerate folder IDs across all folder-capable sources.
   - if any duplicate folder ID appears across sources, abort before namespace creation.
   - emit `vim.notify('dbee: duplicate folder_id detected across sources; migration aborted. Repair sidecars before retry.')`.
   - do NOT write sentinel, delete `notes/global/`, or create backup.
5c. If `notes/global/` exists:
   - collect readable `*.sql` files only.
   - emit `vim.notify('dbee: migrating <N> notes into <F> folders')` before clone.
   - stage all clones into `notes/.phase23-staging-<pid>-<random>/folder:<id>/` directories.
   - NO writes go directly to final `notes/folder:<id>/` paths during clone.
   - clone each global SQL file into each staged folder namespace, preserving basename.
   - if a target filename already exists, preserve existing target and write a collision-safe suffix such as `<stem>.global-migrated.sql`.
5d. Verify all staged paths exist and match the expected file count. If verification fails, `rmrf` the staging dir and abort migration without sentinel/backup/global delete.
5e. Atomically promote staging: `os.rename(staging_dir, final_dir)` per folder namespace. If ANY rename fails mid-promotion, `rmrf` remaining staged dirs AND any successfully-promoted final dirs, then abort without sentinel/backup/global delete.
6. Backup AFTER successful promotion. Create backup by moving/copying legacy global dir only to `notes/global.bak`, `notes/global.bak.YYYYMMDDHHMMSS`, or `notes/.phase23-backups/global.YYYYMMDDHHMMSS`. If backup creation fails, log warning but continue; backup is for safety, not correctness after clones are promoted.
7. AFTER backup, remove the original `global/` path. If delete fails, log warning. Sentinel still gets written because clones are in place.
8. Write sentinel atomically by writing `<sentinel>.tmp` then renaming. If sentinel write fails, the migration is functionally complete (clones in place, global backed up) but sentinel-driven idempotency is broken; emit warning and add `notes/.phase23-recovery-needed`. On next launch, the recovery path detects this marker and writes the missing sentinel.
9. If `notes/global/` does not exist:
   - create current folder namespace dirs.
   - write sentinel.
10. Always release lock on success or failure.
11. Emit completion notification after successful migration.
</action>

Required rename behavior:

1. Keep handler rename behavior.
2. Do not move or rename any `folder:<folder_id>` directory.

Required delete behavior:

1. Update confirmation label from `Delete (members ungrouped)` to mention note deletion, for example `Delete folder and notes`.
2. BOTH delete surfaces MUST route through ONE EditorUI-backed lifecycle helper: `editor:delete_folder_namespace(folder_id)`.
3. Required lifecycle helper behavior:
   - stage folder dir by renaming `notes/folder:<folder_id>/` to `notes/.phase23-trash-<pid>-<random>/folder:<id>/`.
   - if namespace dir does not exist, treat staging as successful no-op.
   - call `handler:source_remove_folder(source_id, folder_id)` from the lifecycle path.
   - if handler removal succeeds, delete the staged trash dir.
   - if handler removal fails, restore the staged namespace directory and show/log the error.
4. Apply the same helper to both:
   - drawer-row action table `delete_folder`
   - bulk-delete action
5. The model/convert layers may pass action metadata and callbacks, but MUST NOT perform direct filesystem operations. The EditorUI lifecycle helper owns note namespace staging/deletion.

Required move behavior:

1. Do not move note files when a connection moves between folders.
2. The picker namespace changes because `Handler:get_folder_for_connection(conn_id)` returns the new folder.
3. Moving to ungrouped makes picker global namespace nil.
