# 23-02 - Migration And Folder Namespace Lifecycle

<objective>
Migrate legacy global notes into folder namespaces once, prevent `global` recreation, and make folder create/delete/rename/move operations maintain note namespace semantics.
</objective>

<threat_model>
- Risk: concurrent nvim instances both migrate and delete/copy the same files. Mitigation: atomic lock directory plus post-lock sentinel recheck.
- Risk: folder deletion permanently deletes notes before folder removal fails. Mitigation: stage namespace directory and restore on handler failure.
- Risk: unsafe folder IDs from manual sidecar edits escape notes root. Mitigation: validate folder IDs before namespace filesystem operations.
- Risk: migration partial failure deletes legacy global. Mitigation: clone first, backup/rename second, sentinel last.
- Risk: bind mounts or symlinks make rename cross filesystem. Mitigation: same-filesystem preflight and fail-closed `EXDEV` handling; no copy+unlink fallback.
- Risk: stale staging/trash dirs leak deleted or partially migrated notes after a crash. Mitigation: startup GC of stale version-coupled scratch dirs before lock acquisition.
- Risk: configured `editor.directory` diverges from migration path. Mitigation: resolve `notes_dir` in `setup_handler()` from the same config/default used by `EditorUI:new()` and pass it into the migration helper.
- Risk: crash mid-promote loses the in-memory rollback list. Mitigation: persist `.notes-migration-v1.promote-manifest` incrementally with actual final paths before each rename.
</threat_model>

<must_haves>
- Migration is first-launch post-Phase-23 and idempotent.
- Migration clones, not moves, each legacy global note into every existing folder namespace.
- Original `notes/global/` path is removed after successful clone/backup.
- Backup remains available until manual cleanup.
- Backup path is canonical: `notes/global.bak`, or `notes/global.bak.YYYYMMDDHHMMSS` only if the default backup already exists.
- Folder created after sentinel gets an empty namespace only.
- Folder rename never moves a namespace directory.
- Folder delete confirmation mentions note deletion.
- Editor startup never calls `namespace_create_note("global", "welcome")` after Phase 23 migration.
</must_haves>

   - If `age_seconds < 300`, treat as fresh lock and fail fast with concurrent-instance message.

   - If total folder count is zero and `notes/global/` has readable `*.sql` files, copy global to the canonical backup BEFORE delete. If backup CREATION fails for any reason (including ENOSPC, EACCES, or EXDEV), ABORT migration: do NOT delete `notes/global/`, do NOT write `.notes-migration-v1`, and emit `vim.notify('dbee: backup of legacy global notes failed; migration aborted to prevent data loss', vim.log.levels.ERROR)`. This is different from the staged-promote path because no clones exist; backup IS the data. 5b. Duplicate folder ID guard:
   - enumerate folder IDs across all folder-capable sources.
   - if any duplicate folder ID appears across sources, abort before namespace creation.
   - emit `vim.notify('dbee: duplicate folder_id detected across sources; migration aborted. Repair sidecars before retry.')`.
   - do NOT write sentinel, delete `notes/global/`, or create backup.
5c. If `notes/global/` exists:
   - collect readable `*.sql` files only.
   - emit `vim.notify('dbee: migrating <N> notes into <F> folders')` before clone.
   - stage all clones into `notes/.notes-migration-v1.staging-<pid>-<random>/folder:<id>/` directories.
   - NO writes go directly to final `notes/folder:<id>/` paths during clone.
   - clone each global SQL file into each staged folder namespace, preserving basename.
   - if a target filename already exists, preserve existing target and prepare a collision-safe suffix such as `<stem>.global-migrated.sql`.
5d. Verify all staged paths exist and match the expected file count. If verification fails, remove the staging dir with `notes_namespace.recursive_rmdir(path)` and abort migration without sentinel/backup/global delete.
5e. Promote staging per file:
   - For each folder namespace, ensure final dir exists with `vim.loop.fs_mkdir` if missing.
   - For each staged file, compute final path. If final path exists, use `<stem>.global-migrated.sql` and add a numeric suffix if needed.
   - Maintain `promote_rollback_paths`, an in-memory list of `dst_final_path` values created by this process for rollback on same-process promote failure.
   - Maintain `notes/.notes-migration-v1.promote-manifest`, an on-disk persistent JSON object with `expected_count`, `promote_complete`, and an `entries` list of `{src_staging_path, dst_final_path}` pairs where `dst_final_path` is the ACTUAL path written after collision suffixing. Before each per-file rename, atomically write the next manifest state through tmp+rename with `promote_complete = false`. After the rename succeeds, read back the on-disk file and confirm the last-written entry is present. After all per-file renames succeed and entry count equals `expected_count`, atomically rewrite the manifest with `promote_complete = true`.
   - Rename staged file to final file with `vim.loop.fs_rename`.
   - Append every successfully-created `dst_final_path` to `promote_rollback_paths`.
   - If `fs_rename` returns `EXDEV`, abort with the GN-20 notification and do not attempt copy+unlink fallback.
   - On any non-EXDEV per-file rename failure, rollback by deleting only files in `promote_rollback_paths`. Do NOT delete the entire final namespace dir because it may contain user-created folder notes.
6. Backup AFTER successful promotion. Create backup by moving/copying legacy global dir only to `notes/global.bak`, or `notes/global.bak.YYYYMMDDHHMMSS` if `global.bak` already exists. If backup creation fails, log warning but continue; backup is for safety, not correctness after clones are promoted. This non-fatal rule applies only after staged promotion succeeds, never to the zero-folder-with-content branch.
7. AFTER backup, remove the original `global/` path. If delete fails, log warning. Sentinel still gets written because clones are in place.
8. Write sentinel atomically by writing `<sentinel>.tmp` then renaming. If sentinel write fails, the migration is functionally complete but sentinel-driven idempotency is broken; emit warning and write `notes/.notes-migration-v1.recovery-needed` containing a recovery-manifest JSON. This recovery-manifest is a SEPARATE artifact from the promote-manifest and is written ONLY on sentinel-write failure after promote success:
   ```json
   {
     "folder_ids": ["folder_X", "folder_Y"],
     "final_paths": ["/notes/folder:folder_X/welcome.sql", "/notes/folder:folder_Y/welcome.global-migrated.sql"],
     "backup_path": "notes/global.bak",
     "global_deleted": true,
     "timestamp": "<ISO-8601>"
   }
   ```
   Recovery on next launch runs post-lock and post-sentinel-recheck. It reads the recovery-manifest, verifies each exact `final_paths[i]` file exists with non-zero size, writes `.notes-migration-v1`, then removes the recovery-manifest. If verification fails, log an error, notify the user, and retain the recovery-manifest for intervention.
   After a normal sentinel write succeeds, delete `.notes-migration-v1.promote-manifest`.
9. If `notes/global/` does not exist:
   - create current folder namespace dirs.
   - write sentinel.
10. Always release lock on success or failure.
11. Emit completion notification after successful migration.
