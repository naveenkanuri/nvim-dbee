# 17-05 - Drawer Label Rendering For PostgreSQL Fields

<objective>
Render PostgreSQL-specific rich metadata annotations in existing drawer node labels and enforce the PostgreSQL view-vs-materialized-view index-folder topology without changing FK navigation behavior.
</objective>

<must_haves>
- Preserve existing label order for `[PK]`, `[NOT NULL]`, and `[FK...]`.
- Add `[GEN]`, `[IDENTITY]`, and `[DEFAULT=...]` for columns.
- Add `[INCLUDE col_a, col_b]` for indexes, capped at five INCLUDE columns plus a `+N more` suffix and per-name display-width truncation.
- Keep `pk_backed=true` hidden through existing drawer behavior.
- In `_build_rich_table_children`, add the Indexes folder only with an explicitly parenthesized materialization gate: `support.indexes == true and (struct.type == "table" or struct.type == "materialized_view")`; regular `struct.type == "view"` nodes keep Columns only.
</must_haves>

<threat_model>
The main risks are label bloat or malformed tags from PostgreSQL type/default punctuation, and a topology regression where regular views render a spurious Indexes folder after PostgreSQL support reports `indexes=true`. Rendering must be deterministic and safe for strings containing parentheses, commas, and brackets; view/MV folder behavior must be materialization-gated.
</threat_model>

<action>
Update `column_label(column)` in `lua/dbee/ui/drawer/convert.lua` with this deterministic tag behavior:

1. If `column.primary_key == true`, append `PK`.
2. If `column.nullable == false`, append `NOT NULL`.
3. If `column.generated` is a non-empty string, append `GEN`.
4. If `column.identity` is a non-empty string, append `IDENTITY`.
5. If `column.default` is a non-empty string and `column.generated` is empty, append `DEFAULT=<expr>`.
6. For default expressions, trim leading/trailing whitespace, collapse internal whitespace to one space, and cap the displayed expression at 80 display characters using `vim.fn.strcharpart(expr, 0, 77) .. "..."` when truncation is needed. Do not use byte slicing because it can corrupt UTF-8.
7. Preserve existing FK label behavior after these tags.

The final label order must be:

```text
name   [type] [PK] [NOT NULL] [GEN] [IDENTITY] [DEFAULT=expr] [FK...]
```
</action>

Update `M.index_nodes(parent_id, indexes)` in `lua/dbee/ui/drawer/convert.lua`:

1. Keep current key-column rendering from `index.columns` and `index.orders`.
2. If `index.include_columns` is a non-empty table, append a separate tag formatted as `INCLUDE col_a, col_b`.
3. Do not merge `include_columns` into the key-column list.
4. Use the first five entries from `index.include_columns` in slice order, which is already sorted by `column_position` in SQL. Do not re-sort in Lua.
5. Cap each rendered INCLUDE name at 30 display columns using `vim.fn.strdisplaywidth(name)`. Names wider than 30 display columns render as the first UTF-8-safe 27 display columns plus `...`.
6. If more than five INCLUDE columns are present, append `+N more` inside the same INCLUDE tag.
7. Preserve `UNIQUE` rendering and existing node metadata.
</action>
