# 17-02 - PostgreSQL Rich Metadata Adapter

<objective>
Implement PostgreSQL rich metadata optional interfaces with PostgreSQL 12+ `pg_catalog` SQL, positional binds, composite PK/FK ordinal pairing, INCLUDE-index separation, and schema-level sequence support.
</objective>

<must_haves>
- PostgreSQL only.
- New production logic lives in `dbee/adapters/postgres_driver_rich_metadata.go`.
- Use the five `pg_catalog` SQL shapes from `17-RESEARCH-CODEX.md`, with the `pg_get_serial_sequence` expression revised to the guarded Phase-17 shape below.
- Use `QueryWithArgs(ctx, sql, opts.Schema, opts.Table)` and `QueryWithArgs(ctx, sql, schema)`.
- Do not use `sql.Named`.
</must_haves>

<threat_model>
SQL injection risk is controlled by `$1/$2` positional binds. Metadata correctness risk is controlled by catalog scoping on `pg_namespace.nspname` and `pg_class.relname`. PostgreSQL 10/11 are below the floor: `SupportsRichMetadata()` stays true for PostgreSQL, and below-floor servers fail loud from rich metadata load errors instead of adding a capability-probe fallback.
</threat_model>

Required PostgreSQL-specific adjustments:

1. `postgresColumnsRichSQL` MUST use `c.relkind IN ('r','p','f','v','m')`. Plain views are included because Phase 16 drawer paths already prefetch view columns; partitioned and foreign tables are rendered as ordinary table-like relations for Phase 17.
2. `postgresIndexesSQL` MUST include materialized views and partitioned tables but MUST exclude foreign tables because they do not own `pg_index` rows.
3. `serial_sequence` MUST avoid an unconditional per-row `pg_get_serial_sequence(...)` call. Use the exact CTE-based `postgresColumnsRichSQL` shape below so `default_expr` is produced in an outer scope before the guarded CASE references it.
4. Use `pg_get_expr`, not deprecated `pg_attrdef.adsrc`.
5. Foreign tables (`relkind='f'`) remain labeled as `table` by existing `postgres_driver.go` topology. Phase 17 does not relabel them; they get Columns rich metadata like regular tables. Because PostgreSQL does not expose `pg_index` rows for foreign tables, an empty Indexes folder for a foreign table is acceptable and does not need a separate strict sentinel.

```go
const postgresColumnsRichSQL = `
	WITH cols AS (
	  SELECT n.nspname AS schema_name,
	         c.relname AS table_name,
	         a.attname AS column_name,
	         pg_catalog.format_type(a.atttypid, a.atttypmod) AS data_type,
	         NOT a.attnotnull AS nullable,
	         a.attgenerated,
	         a.attidentity,
	         pg_catalog.pg_get_expr(d.adbin, d.adrelid, false) AS default_expr,
	         a.attnum
	  FROM pg_catalog.pg_class c
	  JOIN pg_catalog.pg_namespace n
	    ON n.oid = c.relnamespace
	  JOIN pg_catalog.pg_attribute a
	    ON a.attrelid = c.oid
	  LEFT JOIN pg_catalog.pg_attrdef d
	    ON d.adrelid = c.oid
	   AND d.adnum = a.attnum
	  WHERE n.nspname = $1
	    AND c.relname = $2
	    AND c.relkind IN ('r', 'p', 'f', 'v', 'm')
	    AND a.attnum > 0
	    AND NOT a.attisdropped
	)
	SELECT column_name,
	       data_type,
	       nullable,
	       attgenerated,
	       attidentity,
	       default_expr,
	       CASE
	         WHEN default_expr LIKE 'nextval(%' THEN
	           pg_catalog.pg_get_serial_sequence(
	             pg_catalog.quote_ident(schema_name) || '.' || pg_catalog.quote_ident(table_name),
	             column_name
	           )
	         ELSE NULL
	       END AS serial_sequence
	FROM cols
	ORDER BY attnum`
```
