# 17-03 - Materialized View Table-Like Handling

<objective>
Relabel PostgreSQL materialized views as `materialized_view` and ensure drawer searchable/table-like paths plus LSP metadata/hover/completion paths treat MVs consistently. Wave 5 owns the regular-view no-indexes folder gate.
</objective>

<must_haves>
- Change both PostgreSQL structure query paths from `'VIEW'` to `'MATERIALIZED VIEW'`.
- Add `materialized_view = true` to drawer `SEARCHABLE_TYPES` so MV nodes reach structure decoration.
- Add `materialized_view = true` to `TABLE_LIKE_TYPES`.
- Add `materialized_view` to drawer refresh expansion-capture exclusions so MV expansion replay matches table/view replay.
- Update PostgreSQL LSP metadata fallback SQL so it returns base/partitioned/foreign tables as `table`, regular views as `view`, and materialized views as `materialized_view`.
- Add `materialized_view` to every LSP schema-cache/server materialization gate, not only the `MATERIALIZATIONS` constant.
- Update LSP hover and completion formatting so materialized views render as "Materialized View" and use the same completion kind as views.
- Bump `SCHEMA_CACHE_VERSION` or implement explicit stale-cache invalidation so old view-only cached metadata cannot silently preserve stale MV labels.
- Update `convert.decorate_structure_node` early-return and legacy column branch for `materialized_view`.
- Add `materialized_view = true` to drawer `yankable_types` so `yank_name` works for materialized-view rows.
- Add a reusable `make perf-headless ARGS='...'` target backed by `PERF_NVIM_HEADLESS` before any headless rich-metadata verification runs.
- Create the PostgreSQL headless marker scaffold here so Wave 5 can add label and regular-view no-index assertions without creating the same file.
- Do not edit locked schema/epoch helpers.
</must_haves>

<threat_model>
The main risk is a silent UI regression where MVs get the correct backend type but lose lazy children, refresh replay parity, LSP probe resolution, hover/completion labeling, metadata fallback presence, or helper actions. This wave must cover searchable admission, drawer init table-like checks, refresh expansion capture, LSP materialization probing, LSP fallback metadata classification, and convert decoration gates.
</threat_model>

```sql
SELECT n.nspname AS schema_name,
       c.relname AS table_name,
       CASE c.relkind
         WHEN 'r' THEN 'table'
         WHEN 'p' THEN 'table'
         WHEN 'f' THEN 'table'
         WHEN 'v' THEN 'view'
         WHEN 'm' THEN 'materialized_view'
       END AS obj_type
FROM pg_catalog.pg_class c
JOIN pg_catalog.pg_namespace n ON n.oid = c.relnamespace
WHERE c.relkind IN ('r', 'p', 'f', 'v', 'm')
  AND n.nspname NOT IN ('pg_catalog', 'information_schema')
ORDER BY 1, 2
```

If implementation chooses a `pg_matviews` union instead, it MUST still classify partitioned tables and foreign tables as `table` and materialized views as `materialized_view`.
