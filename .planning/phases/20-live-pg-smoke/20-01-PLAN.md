# 20-01 - Live PostgreSQL Rich Metadata Smoke

<objective>
Create a focused live PostgreSQL smoke lane that runs Phase 17 rich metadata SQL against a real `postgres:16-alpine` container, catches the composite-FK SQL-shape class that sqlmock missed, and exposes deterministic `LIVE_PG20_*` markers through `make live-pg-smoke`.
</objective>

<must_haves>
- Do not modify `dbee/adapters/postgres_driver_rich_metadata.go`.
- Do not modify locked helpers: `lua/dbee/schema_filter_authority.lua`, `lua/dbee/schema_name_canonical.lua`, `lua/dbee/lsp/epoch_authority.lua`.
- Use testcontainers-go and the existing adapter connection path, not hand-written `podman run` / `docker run` lifecycle code.
- Use PG16 as the primary pinned version: `postgres:16-alpine`.
- Keep live-smoke rollup separate from `UX13_ROLLUP_LOG`.
- Sequence assertions cover `Name`, `Schema`, `Increment`, and `CacheSize` only. Do not add or assert `current_value`.
</must_haves>

<strict_markers>
`LIVE_PG20_RUNTIME_DETECTED_OK`, `LIVE_PG20_CONTAINER_READY_OK`, `LIVE_PG20_SEED_OK`, `LIVE_PG20_SUPPORT_OK`, `LIVE_PG20_COLUMNS_RICH_OK`, `LIVE_PG20_COMPOSITE_PK_OK`, `LIVE_PG20_FK_COMPOSITE_OK`, `LIVE_PG20_ROWS_FROM_SQL_SHAPE_OK`, `LIVE_PG20_HISTORICAL_UNNEST_NEGATIVE_OK`, `LIVE_PG20_INDEXES_OK`, `LIVE_PG20_MV_INDEXES_OK`, `LIVE_PG20_VIEW_NO_INDEXES_OK`, `LIVE_PG20_SEQUENCE_OK`, `LIVE_PG20_MULTI_SCHEMA_OK`, `LIVE_PG20_SCHEMA_SCOPE_OK`, `LIVE_PG20_SNAPSHOT_OK`, `LIVE_PG20_LOCKED_HELPERS_UNTOUCHED_OK`, `LIVE_PG20_STRICT_MARKER_COUNT`, `PHASE20_ALL_PASS`.
</strict_markers>

<threat_model>
The live lane is intentionally environment-gated. Main risks are runtime variance, slow image pulls, false skips, nondeterministic catalog ordering, and accidental helper/prod-SQL churn. The plan mitigates these by health-checking podman/docker before test launch, failing CI through `LIVE_PG20_REQUIRED=1`, sorting canonical snapshot data, pinning PG16, and adding grep/git-diff gates before `PHASE20_ALL_PASS`.
</threat_model>

   - Include at least one regular index with DESC order and INCLUDE columns.
   - Include `pg20_analytics.active_customers` as a regular view.
   - Include `pg20_analytics.customer_order_summary` as a materialized view with at least one index and one INCLUDE index.
   - Include at least one sequence in `pg20_sales` and one in `pg20_inventory` with deterministic `INCREMENT BY` and `CACHE` values.
   - Include similarly named objects in multiple schemas to support schema-scope assertions.

Add `dbee/tests/integration/postgres_rich_metadata_smoke_test.go` with top-level test `TestPostgresLiveRichMetadataSmoke`.

The test must:
1. Start one shared rich metadata PostgreSQL container and clean it up with `tc.CleanupContainer`.
2. Verify fixture existence by querying seed objects, then log `LIVE_PG20_CONTAINER_READY_OK=true` and `LIVE_PG20_SEED_OK=true`.
3. Call `suite.d.SupportsRichMetadata()` or equivalent public connection method and assert all three booleans are true; log `LIVE_PG20_SUPPORT_OK=true`.
4. Call `GetColumnsRich(&core.TableOptions{Schema: "pg20_sales", Table: "order_items", Materialization: core.StructureTypeTable})`:
   - Assert columns include `tenant_id`, `order_id`, `line_no`, and fields expected from generated/default/identity fixtures where applicable.
   - Assert composite PK ordinals are `tenant_id=1`, `order_id=2`, `line_no=3`; log `LIVE_PG20_COMPOSITE_PK_OK=true`.
   - Assert composite FK refs include full `SourceColumns` and `TargetColumns` arrays in ordinal order; log `LIVE_PG20_FK_COMPOSITE_OK=true`.
   - Log `LIVE_PG20_ROWS_FROM_SQL_SHAPE_OK=true` only after this public method succeeds against live PostgreSQL.
   - Log `LIVE_PG20_COLUMNS_RICH_OK=true` after generated/default/identity/nullability assertions pass.
5. Run the historical malformed SQL shape through a raw `database/sql` connection to the same container:
   - Query text must include `pg_catalog.unnest(ARRAY[1::smallint], ARRAY[1::smallint]) WITH ORDINALITY` or the exact historical two-array `pg_catalog.unnest` class.
   - Assert PostgreSQL returns an error.
   - Log `LIVE_PG20_HISTORICAL_UNNEST_NEGATIVE_OK=true`.
6. Call `GetIndexes` for a regular table and assert ordered key columns, DESC order, uniqueness/PK-backed state, and INCLUDE separation; log `LIVE_PG20_INDEXES_OK=true`.
7. Call `GetIndexes` for `pg20_analytics.customer_order_summary` and assert the materialized-view index appears; log `LIVE_PG20_MV_INDEXES_OK=true`.
8. Call `GetIndexes` for `pg20_analytics.active_customers` and assert it returns an empty slice without error; log `LIVE_PG20_VIEW_NO_INDEXES_OK=true`.
9. Call `GetSequences("pg20_sales")` and `GetSequences("pg20_inventory")`; assert `Name`, `Schema`, `Increment`, and `CacheSize`; log `LIVE_PG20_SEQUENCE_OK=true`.
10. Assert multiple schemas were used and same/similar object names do not leak across schema-scoped calls; log `LIVE_PG20_MULTI_SCHEMA_OK=true` and `LIVE_PG20_SCHEMA_SCOPE_OK=true`.
11. Build a canonical sorted snapshot containing selected columns, FKs, indexes, and sequences. Compare it to either an inline golden string or `dbee/tests/testdata/postgres_rich_metadata_snapshot.json`; log `LIVE_PG20_SNAPSHOT_OK=true`.
12. Do not import or call Lua/nvim surfaces.
</action>
