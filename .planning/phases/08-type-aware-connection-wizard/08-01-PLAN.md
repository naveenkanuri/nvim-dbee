<constraints>
- Honor Phase 4 D-31, Phase 5 D-29, Phase 6 D-30 through D-63, Phase 7 D-64 through D-88, and Phase 8 D-97 through D-100 verbatim.
- Do not widen `dbee/core/connection_params.go`, `connection_get_params()`, or `source_get_connections()` return shapes.
- FileSource metadata stays source-local. The persisted record may grow an additive `wizard` block, but runtime connections still expose only `id`, `name`, `type`, and `url`.
- `source_add_connection()` and `source_update_connection()` keep their existing function names and argument order. Later plans may pass additive keys in the `details` table, but this plan must not rename or replace those surfaces.
- Atomic writes must use a temp file in the same directory as the target JSON, then rename into place.
- Unknown field preservation is semantic only. Preserve untouched nested keys inside additive metadata blocks such as `wizard`, but do not promise comment, ordering, or whitespace preservation across rewrites.
- Explicit deletion uses a reserved control field such as `details.__remove_keys = { "wizard" }`. FileSource consumes that control field, removes those top-level keys before recursive merge, and never persists the control field itself.
- Raw-record helper failure must be non-fatal: sources that cannot provide raw records return `nil` and later plans fall back to parse/raw edit seeding.
</constraints>

```lua
handler:source_get_connection_record(source_id, conn_id) -> table|nil
```

FileSource write rule:
```lua
read target file -> decode raw records -> mutate target record(s) in memory
-> encode JSON -> write temp file in same directory -> close temp file
-> rename temp file over target
```

FileSource update delete contract:
```lua
source:update(id, details)

details = {
  name = "Legacy DSN",
  type = "sqlite",
  url = "sqlite:///tmp/legacy.db",
  __remove_keys = { "wizard" },
}

-- FileSource strips __remove_keys before persistence
-- and deletes the listed top-level keys before recursive merge
```

  <action>
    Rebuild FileSource persistence around raw records instead of lossy connection-field rewrites.

    1. Add raw-record helpers in `lua/dbee/sources.lua`.
       - read all records as raw Lua tables
       - locate records by `id`
       - encode/write records atomically through a same-directory temp file

    2. Refactor `create()`, `update()`, and `delete()` to use those helpers.
       - `create()` appends the full record, including any additive wizard metadata block
       - `update()` overlays changed fields onto the existing record instead of rebuilding from only `name`, `type`, and `url`
       - explicit control such as `details.__remove_keys = { "wizard" }` deletes listed top-level keys before recursive merge so raw-compatibility updates can physically drop stale metadata
       - additive metadata blocks such as `wizard` use recursive merge or explicit carry-forward so untouched nested keys survive
       - `delete()` removes only the targeted record and preserves every sibling entry

    3. Make failure fail closed.
       - JSON encode failure
       - temp file open/write/close failure
       - rename failure
       all leave the original file untouched and raise an actionable error

    4. Preserve D-100 semantics explicitly.
       - unknown sibling records survive untouched
       - unknown fields on the edited record survive unless the new payload intentionally overwrites them or explicitly removes them through `__remove_keys`
       - untouched nested keys inside additive metadata blocks such as `wizard` survive when known wizard leaves are updated
       - do not promise comment or formatting preservation
  </action>

  </verify>

    - FileSource create/update/delete no longer overwrite the target JSON in place.

    - Additive wizard metadata survives create/update round-trips.
    - Failure leaves the original file untouched.
