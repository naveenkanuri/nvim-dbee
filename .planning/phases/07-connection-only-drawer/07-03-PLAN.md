By the end of this plan, they share one authoritative coordination layer for root warmups and invalidation bursts, reconnect ID rewrites remain visually warm in the drawer, and the last synchronous connection-expand seam is absorbed into the same placeholder-and-patch model as Phase 6 async columns.

  <action>
    Centralize same-connection root warmups.

    1. Add handler-owned `authoritative_root_epoch[conn_id]`.
       - initialize missing connections at epoch `0`
       - eventful invalidations consume and increment this handler-owned epoch source
       - drawer mirrors the current handler epoch into `_struct_cache.root_epoch[conn_id]` at the same pre-request-clear moment Phase 6 already owns
       - LSP-first loads read the current handler epoch instead of inventing a consumer-local value

    2. Add a handler-owned single-flight layer keyed by `(conn_id, authoritative_root_epoch)`.
       - record flights and waiters in handler-owned state
       - launch one underlying root fetch for duplicate same-key requests
       - on success or error, fan out the terminal payload, then drop the flight + waiter list
       - on supersession, notify the old waiters with `{ error_kind = "superseded", new_epoch = N }`, drop the old waiter list, and start the new flight
       - on consumer teardown, remove only that consumer's waiter slot; do not add transport cancellation

    3. Preserve the raw full-tree contract.
       - keep the underlying Go `structure_loaded` contract backward compatible
       - if needed, relay waiter-specific completion through an additive handler-local fanout path instead of mutating the raw Go payload

    4. Move drawer and LSP bootstrap/reload entry points onto the shared coordination path.
       - register each consumer's `connection_invalidated` listener before snapshot reads begin
       - handler allocates bootstrap generation `0` for that consumer and starts queueing invalidations into it immediately
       - buffer up to `64` invalidations per consumer between listener registration and the consumer's live-mode flip
       - consumer requests snapshot through the canonical `get_sources()` + `source_get_connections()` + `get_current_connection()` path
       - snapshot includes `snapshot_authoritative_epoch[conn_id]`
       - bootstrap generation `0` remains active and appendable after snapshot completion; invalidations arriving after the snapshot returns but before drain/live flip still queue into generation `0`
       - consumer drains generation `0`, applying only buffered invalidations whose `authoritative_root_epoch` exceeds the snapshot epoch for the affected connection; older events are absorbed by the snapshot and dropped
       - after the ordinary drain pass, consumer calls synchronous `handler:promote_to_live(consumer_id)` to finish bootstrap
       - handler owns promotion as one mutex-protected seal-and-flip operation: stop appending to bootstrap generation `0`, snapshot the buffer tail that arrived since the consumer's last drain read, return that tail synchronously to the consumer, mark the consumer `live`, and destroy bootstrap generation `0`
       - consumer drains the returned tail synchronously before reading from the live listener channel, so events that land before handler seals generation `0` are never stranded in a destroyed buffer
       - invalidations arriving after handler seals generation `0` and flips the consumer to `live` go directly through the live listener path and are processed only after the returned tail finishes, preserving tail-before-live ordering across the handoff
       - when a bootstrap buffer reaches `64`, handler marks that bootstrap generation `overflowed`, emits the snapshot-resync warning together with a new bootstrap generation token, clears the overflowed buffer, and immediately allocates a fresh empty bootstrap buffer for generation `N+1`
       - subsequent live events queue into generation `N+1` immediately, so there is no drop window while the consumer resnapshots
       - consumer treats the overflow warning as nested bootstrap recovery: discard the current snapshot attempt, note generation `N+1`, request a fresh snapshot through the same `get_sources()` + `source_get_connections()` + `get_current_connection()` path, drain generation `N+1` against the fresh `snapshot_authoritative_epoch[conn_id]` map, and then finish bootstrap through the same handler-owned `promote_to_live(consumer_id)` tail-drain handoff
       - if generation `N+1` also overflows, repeat the same nested-bootstrap protocol for `N+2` and later generations
       - if one consumer overflows more than `3` consecutive times, handler emits a hard-error `bootstrap_overflow_storm` warning, stops queueing further bootstrap events for that consumer, and requires consumer teardown plus full reinitialize
       - drawer root warmups after authoritative invalidation and LSP startup/refresh stop issuing duplicate independent warmups for the same key

    5. Keep Phase 6 drawer fences intact.
       - drawer still supplies `caller_token = "drawer"` plus request fences
       - handler fanout returns waiter-specific completion so drawer still applies only matching caller/request/epoch payloads
       - LSP keeps its legacy-compatible payload handling
