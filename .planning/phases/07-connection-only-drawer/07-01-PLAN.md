<objective>
Create the lifecycle foundation Phase 7 needs before the drawer rewrite: additive `connection_invalidated` and `source_reload_failed` events, a split silent-vs-eventful source reload path, and an authoritative snapshot bootstrap helper that future drawer/LSP plans can consume.

Purpose: stop hidden source reload side effects from staying implicit. By the end of this plan, lifecycle mutations have one canonical handler-facing surface, reconnect and public source actions can share the same bookkeeping, and later plans can mount consumers from a snapshot rather than hoping they saw every early event.
</objective>

    1. Extract the current `source_reload()` bookkeeping into an additive internal helper, `_source_reload_silent(source_id, opts)`.
       - gather `retired_conn_ids` before reload
       - reload the source and repopulate `source_conn_lookup`
       - gather `new_conn_ids` after reload
       - capture `current_conn_id_before` and `current_conn_id_after`
       - return enough structured state for wrappers to distinguish full success from committed-mutation / reload-failed partial failure
       - include `authoritative_root_epoch` in the structured result whenever the reload step reaches an authoritative handler state transition
       - compute any old->new rewrite pairs for reconnect/internal callers, but keep those details out of the public event payload

    2. Add additive lifecycle emit helpers in `Handler`.
       - one helper to emit `connection_invalidated`
       - one helper to emit `source_reload_failed`
       - `connection_invalidated` carries the D-71 core fields plus additive `authoritative_root_epoch`
       - both route through `require("dbee.handler.__events").trigger(...)`

    3. Refactor the public wrappers:
       - `source_reload`
       - `source_add_connection`
       - `source_update_connection`
       - `source_remove_connection`
       so they call `_source_reload_silent(...)` and then handle the three lifecycle outcomes explicitly:
       - full success -> emit one authoritative `connection_invalidated` payload
       - mutation-step failure before commit -> emit `source_reload_failed` only
       - committed mutation plus failed `_source_reload_silent(...)` bookkeeping -> emit `connection_invalidated` first, then `source_reload_failed`, both with the same `source_id`

    4. Keep silent flows truly silent.
       - reconnect and later identity-rewrite callers can consume `_source_reload_silent(...)`
       - they must not automatically emit the public failure event

    5. Update `lua/dbee/doc.lua` for the new handler lifecycle surface.
