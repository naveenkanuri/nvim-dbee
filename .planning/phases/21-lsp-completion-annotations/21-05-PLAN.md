# 21-05 - Performance Gates

<objective>
Lock performance budgets for completion-item annotation reads, reverse-FK index construction, and resolve-time reverse-FK lookups.
</objective>

<must_haves>
- Perf tests run in headless Neovim and do not require live DBs.
- Fixtures are deterministic and do not depend on wall-clock sleeps.
- Request paths remain cache-only.
- Perf markers are strict because Phase 21 affects per-keystroke completion UX.
- Report p95 over 100 iterations per scenario, matching Phase 17/23 perf methodology.
- State budgets per entry as well as per scenario so large Oracle-style schemas scale predictably.
- Disk-load startup must not synchronously build the reverse-FK index for large v4 disk payloads.
</must_haves>

<action>
Create `ci/headless/check_lsp21_perf.lua`.

Fixture:

1. Mid fixture: 1,000 tables x 10 columns = 10,000 columns.
2. Mid fixture includes 1,000 FK refs distributed across those tables.
3. Large fixture: 5,000 tables x 50 columns = 250,000 columns.
4. Large fixture includes 50,000 FK refs distributed across those tables.
5. Include at least 100 composite FK groups in the mid fixture and at least 1,000 composite FK groups in the large fixture.
6. Include a 500-column single-table wide-completion fixture.
7. Include sustained churn fixture: store/evict 1,000 source tables whose refs are represented in `reverse_fk_refs_by_source_key`.
8. Include a serialized v4 disk payload for the large fixture to exercise `load_from_disk`.
9. Warm the cache before measurement except for the explicit cold disk-load scenario.
10. Run 100 measurement iterations per scenario and compute p95 from those samples.

Measurements:

1. Completion read p95:
   - call `get_column_completion_items` for hot tables for 100 measured iterations;
   - p95 must be `< 1ms`.
2. Wide-table completion p95:
   - call `get_column_completion_items` for the 500-column table for 100 measured iterations;
   - p95 must be `< 2.5ms` and `<= 5us/column`;
   - verify no-marker columns skip `labelDetails` allocation.
3. Reverse-index build:
   - time `_rebuild_column_indexes()` or equivalent public test hook around the 10k/1k-FK fixture;
   - total must be `< 50ms`;
   - insertion budget is `<= 10us/FK-ref`.
4. Large reverse-index build:
   - time the 250k-column/50k-FK fixture;
   - total must be `<= 500ms`;
   - insertion budget remains `<= 10us/FK-ref`.
5. Eviction churn p95:
   - store/evict 1,000 source tables and measure `_drop_reverse_fk_refs_for_source(source_key)`;
   - p95 per eviction must be `< 0.5ms`;
   - this proves eviction uses the source-forward map rather than scanning the full target map.
6. Resolve lookup p95:
   - call `get_reverse_fk_refs` for FK-target and non-target columns for 100 measured iterations;
   - p95 must be `< 1ms`;
   - the measurement includes read-time sorting of 50 capped refs.
7. Resolve E2E p95:
   - measure `metadata_for + format_referenced_by + serialization` through the completionItem/resolve path;
   - p95 must be `< 5ms`;
   - budget breakdown: reverse-FK cache lookup including sort `< 1ms`, docs formatting/escaping `< 2ms`, LSP serialization `< 2ms`;
   - fixture includes high-fan-in truncation so rendering cost is bounded.
8. Disk-load deferred large p95:
   - load the large v4 disk payload through the real `load_from_disk` path;
   - assert `reverse_fk_refs_by_target_key` and `reverse_fk_refs_by_source_key` remain empty during synchronous load;
   - assert `reverse_fk_index_dirty == true`;
   - assert Phase 21 reverse-FK overhead over baseline column-index load is `< 25ms` p95;
   - assert the deferred/background reverse-FK build later reaches the same 50k-ref result under the existing `<= 500ms` large-build budget.
9. Emit diagnostic numeric markers for measured p95/build values if useful, but keep only the eight strict perf markers counted.
10. Emit non-strict diagnostics:
   - `LSP21_DIAG_REVERSE_INDEX_BUILD_CANONICAL_CALLS_PER_REF`;
   - `LSP21_DIAG_REVERSE_INDEX_BUILD_CAP_CHECK_NS`;
   - `LSP21_DIAG_RESOLVE_MEMO_SIZE`.
