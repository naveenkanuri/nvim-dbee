# 21-02 - Reverse-FK Index

<objective>
Maintain an in-memory reverse-FK index derived from loaded column metadata and expose a fail-closed cache API for resolve-time lookup.
</objective>

<must_haves>
- The index is derived from `self.columns`; it is never written to disk.
- Reads route through `epoch_authority.read_with_freshness`.
- Rebuild and clear through existing column-index lifecycle.
- Maintain both target and source maps so source-table eviction is O(refs-for-source), not O(total reverse refs).
- Split freshness from mutation identity: `reverse_fk_cache_epoch` gates stale cache reads, while `reverse_fk_index_generation` invalidates resolve memo entries after any reverse-index mutation.
- Defer reverse-FK rebuild on v4 disk load so large hot caches do not add synchronous startup cost.
- Do not scan all tables on every resolve hover.
- Do not edit `epoch_authority.lua`.
</must_haves>

<action>
Add in-memory reverse-FK state to `SchemaCache:new` and reset paths.

State:

```lua
reverse_fk_refs_by_target_key = {},
reverse_fk_refs_by_source_key = {},
reverse_fk_cache_epoch = 0,
reverse_fk_index_generation = 0,
reverse_fk_index_size = 0,
reverse_fk_index_overflow = false,
reverse_fk_overflow_warned = false,
reverse_fk_index_dirty = false,
reverse_fk_index_building = false,
```

Caps:

```lua
REVERSE_FK_PER_TARGET_REF_CAP = 50
REVERSE_FK_PER_SOURCE_REF_CAP = 1000
REVERSE_FK_TOTAL_REF_CAP = 50000
```

Reset requirements:

1. `_reset_indexes()` clears reverse-FK state.
2. `_drop_column_index(key)` calls `_drop_reverse_fk_refs_for_source(key)`.
3. `_drop_reverse_fk_refs_for_source(source_key)` uses `reverse_fk_refs_by_source_key[source_key]` to remove only target entries contributed by that source table. It must not walk the entire `reverse_fk_refs_by_target_key` map.
4. `_drop_reverse_fk_refs_for_source(source_key)` increments `reverse_fk_index_generation` when it removes refs or overflow bookkeeping.
5. `_rebuild_column_indexes({ reverse_fk = true })` clears reverse-FK state before looping all cached columns, then rebuilds column completion items, target map, and source map.
6. `_rebuild_column_indexes({ reverse_fk = false, reverse_fk_dirty = true })` is the `load_from_disk` path: rebuild column completion items synchronously, clear reverse-FK maps, set `reverse_fk_index_dirty = true`, stamp `reverse_fk_cache_epoch = epoch_authority.cache_epoch(self)`, increment `reverse_fk_index_generation`, and do not scan FKs into reverse maps during startup.
7. After a successful full reverse-FK rebuild or `_refresh_reverse_fk_refs_for_table(...)`, set `self.reverse_fk_cache_epoch = epoch_authority.cache_epoch(self)`.
8. Increment `reverse_fk_index_generation` on every reverse-index mutation: full build, table refresh, source drop, per-target bucket mutation, overflow-counter mutation, clear, and dirty-state transition.
9. `_reset_indexes()` and fail-closed clear paths set `reverse_fk_cache_epoch = 0`, clear dirty/building flags, and increment `reverse_fk_index_generation`.
10. `invalidate()` and `set_connection()` already call `_reset_indexes`; no extra invalidation branch is needed.
11. `_drop_column_index(key)` is now the sole per-source derived-index eviction hub; future per-source derived indexes must register their drop logic there.
</action>

<action>
Add reverse-FK index writers.

Required helpers:

1. `reverse_fk_target_key(schema, table, column)`:
   - logical shape is `target_schema:target_table:target_column`;
   - implementation may use a safe delimiter such as `"\31"` to avoid collisions.
   - writer and reader both normalize each component with `schema_name_canonical.canonical(name, quoted=true, self.fold_id).canonical` before joining;
   - display fields keep the exact adapter-provided schema/table/column text.
2. `reverse_fk_source_key(schema, table)` returns the existing `table_key(schema, table)` source identity and is the key in `reverse_fk_refs_by_source_key`.
3. `reverse_fk_pair_from_ref(ref)`:
   - precedence 1: if `source_columns`/`target_columns` are arrays and `source_ordinal > 0` is in bounds for both arrays, pair `SourceColumns[source_ordinal]` with `TargetColumns[source_ordinal]`;
   - precedence 2: if `source_column` exactly appears in `source_columns`, use that index to select `target_columns[index]`;
   - precedence 3: if both `source_column` and `target_column` are set, use that shorthand pair as a single-column FK;
   - precedence 4: skip malformed refs and log a diagnostic counter; never index every target column for every source column.
4. `_drop_reverse_fk_refs_for_source(source_key)` removes old refs before a table's columns are replaced using only the source-forward map.
5. `_add_reverse_fk_refs_for_column(schema, table_name, col)` adds entries:
   ```lua
   {
     src_schema = schema,
     src_table = table_name,
     src_col = source_column,
     constraint = ref.constraint_name,
     ordinal = ref.source_ordinal,
     target_schema = ref.target_schema,
     target_table = ref.target_table,
     target_col = target_column,
     source_columns = ref.source_columns,
     target_columns = ref.target_columns,
   }
   ```
6. `_add_reverse_fk_refs_for_column` updates both:
   - `reverse_fk_refs_by_target_key[target_key]` for O(1) target lookup;
   - `reverse_fk_refs_by_source_key[source_key]` with contribution records `{ target_key, ref_id, visible, dropped }` for O(refs-for-source) eviction and accurate overflow-counter decrements.
7. Enforce caps while adding:
   - per-target cap 50 refs; store/count overflow for that target so docs can render `+N more`;
   - per-source cap 1000 refs as a memory backstop only; realistic fixtures must not trip it, and it is not the UX truncation mechanism;
   - total cap 50k refs as last-line defense.
8. `_refresh_reverse_fk_refs_for_table(schema, table_name, cols)` drops then re-adds refs for that source table.
9. Iteration order is deterministic: columns are scanned in `cols` array order and each `col.foreign_keys` list is scanned in adapter-provided array order.
10. Cache canonical source schema/table once per `_refresh_reverse_fk_refs_for_table`, and memoize canonical target schema/table pairs within that refresh; target column canonicalization remains per FK ref. Emit diagnostics for canonical calls/ref and cap-check cost.

Per-target bucket shape:

```lua
reverse_fk_refs_by_target_key[target_key] = {
  refs = {}, -- visible refs, capped at 50
  dropped_count = 0,
  dropped_sources = {}, -- counted map: dropped_sources[source_key] = { count = N, refs = { dropped_ref_summaries } }
}
```

The counted `dropped_sources` map is intentional: a raw set cannot decrement `dropped_count` correctly when one source contributes multiple dropped refs, and storing dropped ref summaries lets eviction promote deterministic dropped refs back into `refs` when visible entries fall below cap.

Integrate `_refresh_reverse_fk_refs_for_table` inside `_update_column_index` or immediately around it from `_store_columns`, preserving one source of truth.

Overflow recovery:

- All mutation entry points touching overflow state are `_add_reverse_fk_refs_for_column`, `_drop_reverse_fk_refs_for_source`, `_refresh_reverse_fk_refs_for_table`, full reverse-FK rebuild, and `_reset_indexes`.
- If a later refresh/rebuild drops below the relevant cap, clear the affected overflow/truncation state.
- On `_drop_reverse_fk_refs_for_source(source_key)`, remove visible refs from each affected target bucket and decrement `dropped_count` by `dropped_sources[source_key].count` for non-visible refs. Remove the source key from `dropped_sources`. If visible refs fall below 50 and dropped refs remain, promote dropped ref summaries in deterministic insertion order until the visible cap is full or no dropped refs remain. When `dropped_count == 0`, clear the overflow flag/truncation metadata for that target.
- Emit at most one `vim.notify("dbee-lsp: reverse-FK index overflow; some referenced-by docs truncated", vim.log.levels.WARN)` per overflow episode; reset the warning flag only after the index returns below cap.
</action>

<action>
Add public cache API:

```lua
SchemaCache:get_reverse_fk_refs(schema, table_name, column_name, opts)
```

Required behavior:

1. Wrap the body in `epoch_authority.read_with_freshness(self, self.handler, self.conn_id, function(check) ... end)`.
2. Inside the freshness closure, first call `self:_fresh_lsp_scope()`. If it returns nil, `authority_unavailable`, or `stale_root_epoch`, return `{}`.
3. Caller passes already-resolved `schema/table/column` from `get_column_metadata`; this API does not perform a second lookup/canonicalization pass. It only canonicalizes those exact values into the fold-aware reverse-FK key.
4. Check `reverse_fk_cache_epoch == check.cache_epoch`; return `{}` when stale/missing.
5. Filter returned refs to source schemas/tables admitted by the current `_fresh_lsp_scope()` authority result so filtered-out source tables do not leak through reverse docs.
6. If `reverse_fk_index_dirty == true`, schedule or join a deferred in-memory reverse-FK rebuild without blocking this resolve request. Return `{}`/unchanged docs for the current request; a later resolve after build completion returns refs.
   - Deferred build captures the starting cache epoch and only installs rebuilt maps if `epoch_authority.cache_epoch(self)` still matches that captured epoch; otherwise discard the build result and keep fail-closed state.
7. Return an owned array sorted on read by `(src_schema, src_table, src_col, constraint, ordinal)`. Sorting at read time is locked because per-target refs are capped at 50, making O(N log N) bounded while avoiding insertion-time sort cost during large builds. `resolve.metadata_for` owns that array and must not deep-copy it again.
8. If a target/source/global cap truncated results, include truncation metadata on the returned array (for example `_truncated_count` and `_overflow = true`) so documentation can render a visible `+N more` line. Do not collapse all lookups to silent `{}` on overflow.
9. Add `reverse_fk_index_size`, `reverse_fk_cache_epoch`, `reverse_fk_index_generation`, dirty/building flags, per-target/source truncation counters, and `reverse_fk_index_overflow` to `get_stats()`.
</action>
