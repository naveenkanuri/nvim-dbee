# 18-01 - Drawer Database Container Topology

<objective>
Add a drawer-only `database` container for adapters with a real database/catalog tier, wrapping schemas/objects under the active database while preserving flat topology for adapters where a database layer would be redundant or misleading.
</objective>

<must_haves>
- Modify only `lua/dbee/ui/drawer/init.lua`, `lua/dbee/ui/drawer/convert.lua`, `lua/dbee/ui/drawer/model.lua`, and `lua/dbee/config.lua`.
- Do not edit locked helpers.
- Do not add `core.StructureTypeDatabase`, new RPC endpoints, or new event names.
- Nested adapters: PostgreSQL, SQL Server/MSSQL, Redshift, Databricks, MongoDB.
- Flat adapters: Oracle, MySQL, ClickHouse, SQLite, DuckDB, BigQuery, Redis, and unknown adapters.
- Replace the visible `database_switch_node` sibling with an active `database` container for nested adapters.
- A single current database must render even when there are no switch alternatives.
- Database wrapper applies to both full-root and `schemas_only` lazy roots.
- Database node ID is stable per connection and does not include the database name.
- The database wrapper is render-only. `_struct_cache.root[conn_id].structures` remains unchanged; schemas stay top-level in cache.
- Top-level `STRUCTURES_KIND` branch state remains keyed by `conn_id`; schema branch IDs use rendered schema node IDs.
- Old flat schema expansion IDs and snapshots replay under the new database parent.
- Database node is searchable and yankable; yank copies only database name.
</must_haves>

<action>
Add a drawer-local adapter topology decision helper in `init.lua`.

Required behavior:

1. Define a drawer-local canonical topology registry whose keys are exactly repo driver filename stems:
   - nested canonical stems: `postgres`, `sqlserver`, `redshift`, `databricks`, `mongo`
   - flat canonical stems: `mysql`, `clickhouse`, `sqlite`, `duck`, `oracle`, `bigquery`, `redis`
   - deferred canonical stems: none for Phase 18
2. Define a separate alias normalization map, for example:
   - `postgresql -> postgres`
   - `pg -> postgres`
   - `mssql -> sqlserver`
   - `mongodb -> mongo`
   - `duckdb -> duck`
3. `CANONICAL_TOPOLOGY[stem]` must use only canonical driver stems and must exactly match `dbee/adapters/<stem>_driver.go` filename stems.
4. `ALIAS_TO_CANONICAL[alias]` must resolve only to existing canonical stems and must not participate directly in the closed-world file-stem comparison.
5. Unknown external adapters that do not correspond to a repo `*_driver.go` file and do not appear in `ALIAS_TO_CANONICAL` fall back to flat topology.
6. Keep this helper drawer-local; do not add core adapter flags or RPC calls.
7. Add a stable database node ID helper based only on connection ID. The exact ID shape is:
   `conn_id .. ID_SEP .. encode_node_segment({ "__database__" })`.
   The ID must not contain the current database name and must not collide with the existing `_database_switch__` suffix.
8. Add or reuse a way to get the current database display name from `_database_switch_state`, including the `loading` state when `current` is already known.
</action>

</verify>

- Alias normalization is separate from canonical topology; aliases resolve to canonical stems and do not pollute the closed-world file-stem check.
- Unknown adapters remain flat.
- Database node ID helper uses `ID_SEP` + encoded `__database__`, does not concatenate the current database name, and cannot equal the old `conn.id .. "_database_switch__"` ID.

<action>
Add refresh replay, snapshot/hydrate ID rewriting, and old flat schema expansion migration.

Implementation requirements:

1. `_capture_container_expansions` must capture `database` nodes.
2. Preserve existing exclusions for `table`, `view`, `materialized_view`, `column`, and `load_more`.
3. Top-level `STRUCTURES_KIND` branch state remains keyed by `conn_id` even when rendered under the database node.
4. Schema-level branch IDs use the actual rendered schema node ID, which gains the DB parent for nested adapters and remains connection-parented for flat adapters.
5. When replaying container expansions after refresh, support all of:
   - new schema ID under the stable database parent
   - old flat schema ID directly under the connection
   - mixed old+new duplicate captured IDs
6. Migration must be deterministic, single-expansion, and in-memory. Do not write user-state migrations to disk in this phase.
7. Apply the DB-aware ID rewrite through the five snapshot/hydrate paths that can preserve stale IDs across filter/refresh:
   - `clone_rendered_snapshot`
   - `snapshot_to_tree_nodes`
   - `searchable_node_to_tree_node`
   - filter-restore snapshot pre-apply before `self.tree:set_nodes(snapshot_to_tree_nodes(...))`
   - search-merge graft paths that call `self.tree:set_nodes(snapshot_to_tree_nodes(graft), path_id)` and `self.tree:set_nodes(snapshot_to_tree_nodes(filtered_snapshot))`
8. `loaded_lazy_ids` is not disk-migrated. Entries that no longer match a rendered ID are pruned by the existing root-drop path. One re-fetch on first manual expansion is acceptable, but replaying an already-cached schema expansion must not issue `connection_get_schema_objects_async` or `connection_get_schema_objects_singleflight`.
9. Phase 11/12 LSP schema cache files are unaffected because they key on connection/schema metadata, not drawer node ID.
10. The migration helper must be O(N) over captured/rendered IDs. Do not implement nested tree scans per captured ID.
</action>
