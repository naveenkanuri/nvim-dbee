# Phase 12.1: LSP Hover + Completion Resolve - Plan

<objective>
Ship the Phase 12.1 read-only LSP feature surface: cache-backed `textDocument/hover` and cache-truthful `completionItem/resolve`.

By the end of this plan, a user with an active dbee LSP client can hover schema/table/column identifiers in SQL buffers and receive bounded MarkupContent documentation from the existing schema cache, and standard LSP completion clients can resolve dbee schema/table/column completion items for additional documentation. Both surfaces fail closed under schema-filter authority failure, preserve Phase 11 r6 quoted/canonical semantics, do not run sync or async DB metadata work on request paths, and are covered by strict LSP12 sentinels plus existing UX13/ARCH14 rollups.
</objective>

## Upstream Decisions

This plan inherits D-306..D-337 from `.planning/DISCUSS-phase12.md`, D-338..D-365 from the initial Phase 12.1 plan, and D-366..D-377 from Revision 1. Revision 2 kept those decisions and added D-378..D-383. Revision 3 records implementation-fix decisions D-384..D-386.

## Revised Phase 12.1 Decisions

- **D-338:** Phase 12.1 ships only `textDocument/hover` and `completionItem/resolve`. Code actions and symbols remain Phase 12.2/12.3 work.
- **D-339:** Phase 12.1 uses standard LSP protocol methods only; no dbee-specific client or completion plugin protocol is introduced.
- **D-340:** Add rollback flags `lsp.hover` and `lsp.resolve`, both defaulting to `true`. `lsp.hover` intentionally supersedes the earlier longer hover flag placeholder; DISCUSS and RESEARCH artifacts must use `lsp.hover` after this revision.
- **D-341:** `hoverProvider=true` and `completionProvider.resolveProvider=true` are advertised only when the corresponding feature flag is enabled and handler code is loaded.
- **D-342:** Hover is cache-only. It must not call sync handler APIs, schedule async metadata fetches, or touch DB-facing RPCs.
- **D-343:** Hover returns `nil` for SQL keywords, whitespace, punctuation-only cursor positions, unknown identifiers, cache misses, filtered objects, and authority failure.
- **D-344:** Hover responses include the narrow token `range` whenever a token is identified.
- **D-345:** Resolve enriches only dbee object completion items with `data.source == "dbee"` and `data.version == 1`.
- **D-346:** Stale or cache-miss resolve returns protocol-safe incomplete state: no object documentation is attached, no DB fetch is started, and `data.dbee_resolve_status = "incomplete"` is set for tests/trace. LSP `CompletionItem` has no top-level `isIncomplete`.
- **D-347:** Resolve is idempotent. Resolving an already-resolved item returns equivalent docs and does not duplicate or corrupt fields.
- **D-348:** Completion item `data` contains only compact object identity, cache identity/generation, and quote/canonical fields. It must not contain connection params, URLs, usernames, passwords, SQL text, or source config.
- **D-349:** `SchemaCache` exposes a monotonic metadata generation or equivalent cache epoch and it must be coherent with completion item data. Fresh completion items from the current cache generation must resolve successfully; older items must not.
- **D-350:** Hover and resolve read `schema_filter_authority` fresh on every request through a cache wrapper or helper-routed path.
- **D-351:** `authority_unavailable` fails closed for hover/resolve; `api_absent_legacy` preserves legacy implicit-all behavior.
- **D-352:** Hover/resolve object lookup must use quote-aware context data and existing canonical-aware cache lookup methods. No new local fold logic is allowed.
- **D-353:** Markdown output escapes raw schema/table/column names before embedding them in markdown.
- **D-354:** Markdown `MarkupContent` is the default. Plaintext is used when client capabilities explicitly exclude markdown.
- **D-355:** PK/FK/null/default/row-count fields are displayed only when present in cached metadata. Current core column payload guarantees only `name` and `type`.
- **D-356:** Shared object documentation formatting lives in `lua/dbee/lsp/object_docs.lua` to keep hover and resolve behavior identical.
- **D-357:** Existing completion labels, insert text, trigger characters, sort order, and async `CompletionList.isIncomplete` behavior remain unchanged except for additive object `data` on unambiguous dbee object items.
- **D-358:** Hover P95 target is <50ms from cache. Resolve P95 target is <100ms from cache.
- **D-359:** Phase 12.1 perf thresholds are advisory initially, matching D-333, but sentinel markers must be emitted.
- **D-360:** LSP12 headless tests must assert response shape, authority behavior, canonical/quoted lookup, markdown escaping, stale resolve safety, no sync DB work, and no async DB work.
- **D-361:** Add a strict LSP12 rollup and wire it into `make perf-lsp`; also preserve UX13 and ARCH14 rollups.
- **D-362:** No multi-client/per-buffer LSP architecture changes are allowed in Phase 12.1.
- **D-363:** No external SQL parser, formatter, markdown renderer, or new runtime dependency is introduced.
- **D-364:** Cache absence is a clean no-op: hover returns nil, resolve returns original/incomplete item, completion remains usable.
- **D-365:** User documentation and help text are updated only for new feature flags and high-level hover/resolve behavior.
- **D-366:** The executable plan source for Phase 12.1 is `.planning/phases/12-lsp-feature-gap-closure/12-01-PLAN.md`; execute via `$gsd-execute-phase 12`.
- **D-367:** Metadata generation ordering is explicit: either assign the next generation before rebuilding any completion indexes, or attach current generation lazily at `get_*_completion_items()` copy time. The implementation must choose one path and prove fresh returned items resolve.
- **D-368:** Cache-origin completion data represents exact stored object identity. Stored schema/table display names must resolve exact, using `schema_quoted=true` / `table_quoted=true` or explicit exact/canonical keys.
- **D-369:** Global all-column fallback completions may carry dbee resolve data only when the column label maps to exactly one schema/table. Duplicate labels across loaded tables must pass through unresolved.
- **D-370:** Resolve has a dedicated authority fail-closed sentinel independent of stale generation tests.
- **D-371:** Schema completion resolve is in scope and must have a dedicated sentinel.
- **D-372:** Hover/resolve tests must instrument both sync and async metadata APIs. Counters for `connection_get_columns_async`, `connection_get_schema_objects_singleflight`, and equivalent async metadata APIs must stay zero during hover/resolve.
- **D-373:** LSP12 perf probes live outside the existing LSP01 scenario registry. `LSP01_SCENARIOS_COUNT` remains unchanged; LSP12 emits its own `LSP12_PERF_SCENARIOS_COUNT`.
- **D-374:** Hover context scanning is bounded. The hover path scans at most 200 lines or an equivalent bounded byte budget before returning nil on ambiguity.
- **D-375:** Resolve documentation memoization is required. Cache key is exact cache-origin identity, never folded canonical identity: `(kind, exact_schema, exact_table, exact_column, schema_quoted, table_quoted, column_quoted, cache_generation, markdown_kind)`.
- **D-376:** Completion payload growth is bounded: dbee `data` is compact fixed-field identity only, with no docs, previews, column lists, SQL, or connection params.
- **D-377:** LSP12 rollup requires every Phase 12.1 sentinel exactly once unless explicitly documented as a perf numeric marker.
- **D-378:** Resolve memoization must not use `canonical_path` fallback. Exact-distinct objects that share a folded canonical path must resolve and memoize independently in both call orders.
- **D-379:** Generation correctness must be proven for every metadata mutation path that can affect hover/resolve docs. For each path, a pre-mutation item must resolve incomplete/no docs and a post-mutation fresh item must resolve with docs.
- **D-380:** Hover statement extraction scans both backward and forward from the cursor to statement delimiters under one shared 200-line/equivalent byte budget. Bare SELECT-list columns resolve only when exactly one table context is obvious.
- **D-381:** Resolve fail-closed branches scrub dbee-added `documentation` and `detail` fields from already-resolved dbee items before returning incomplete/no-docs state. Non-dbee passthrough remains unchanged.
- **D-382:** No-async static guards are call-path-specific. They check `hover.lua`, `resolve.lua`, and the new metadata snapshot helper bodies only; existing async completion APIs elsewhere in `schema_cache.lua` are not broad-grep failures.
- **D-383:** `LSP12_PERF_SCENARIOS_COUNT=8` is locked for Phase 12.1. Scenario count changes require an explicit D-ledger update and sentinel value bump, matching the UX13 `LSP01_SCENARIOS_COUNT=33` pattern.
- **D-384:** Schema-index and column disk-cache payloads persist `root_epoch`. Rootless legacy schema indexes and column files fail closed instead of being stamped with the current authoritative epoch.
- **D-385:** Hover/resolve may serve docs only when cache metadata root epoch matches the handler authoritative root epoch. Completion resolve data is stamped from cache-owned metadata epoch, not live handler epoch.
- **D-386:** Column loads are column-only mutations. They bump cache generation for stale completion-item detection, but must not advance cache-wide root/schema-index epoch; callbacks whose payload epoch differs from metadata root epoch are rejected.
</decision_ledger>

### Task 7 - Resolve Handler

Implementation notes:

- New `lua/dbee/lsp/resolve.lua`.
- Public API:

```lua
M.handle(item, cache, opts)
```

- Required per-generation memoization:
  - cache key = `kind .. "|" .. exact_schema .. "|" .. exact_table .. "|" .. exact_column .. "|" .. tostring(schema_quoted) .. "|" .. tostring(table_quoted) .. "|" .. tostring(column_quoted) .. "|" .. cache_generation .. "|" .. markdown_kind`
  - never use `canonical_path` as a memo key component or fallback
  - memo is server/cache-session local only
  - any generation/root epoch/cache identity mismatch bypasses memo and returns incomplete/no docs
  - memo entries from prior generations are ignored by key and may be cleared opportunistically
- Flow:
  1. Check `lsp.resolve`.
  2. Validate `item.data.source == "dbee"` and `version == 1`.
  3. Return original for non-dbee items.
  4. Compare cache identity/generation/root epoch.
  5. On stale/miss/ambiguous identity, return item with `data.dbee_resolve_status = "incomplete"` and no object docs.
  6. Read authority fresh and fail closed.
  7. Resolve schema/table/column metadata from cache only using exact/canonical identity from item data.
  8. Format documentation/detail.
  9. Preserve insert behavior fields.
- Fail-closed branches:
  - stale generation/root epoch
  - cache miss
  - authority unavailable
  - object out of scope
  - ambiguous global column identity
  - Each branch must remove dbee-added `documentation` and `detail` before returning the item.
  - Scrubbing applies only to dbee-resolved docs; non-dbee items remain true passthrough.

Acceptance criteria:

- Schema, table, and column docs populate for valid cache hits.
- Non-dbee item passes through.
- Stale item does not receive stale docs and is marked incomplete in `data`.
- Authority-unavailable item receives no docs even when data is valid/non-stale.
- Duplicate global column item receives no docs.
